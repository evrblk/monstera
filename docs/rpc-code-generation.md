# RPC Code Generation

**Explicit code and no magic** - this is one of the core principles of Monstera. The goal was to achieve type-safety and 
utilize compile-time checks without reflection, but eliminate human mistakes from vast boilerplate code at the same time.
That is why all boilerplate RPC code is generated by the framework.

`monstera.yaml` defines all application cores and their operations. The following command runs codegen (from the same
directory where the YAML file is located):

```shell
go tool github.com/evrblk/monstera/cmd/monstera code generate
```

Codegen produces three files into the same directory:

* `api.go` with interfaces for application cores and stubs, plus the request/response type aliases (see below).
* `adapters.go` with adapters to application cores, that turn binary blobs into method calls on a core.
* `stubs.go` with service stubs, that turn requests into binary blobs and route them to the correct application core.

Let's take a look at the example `monstera.yaml` from [Grackle](https://github.com/evrblk/grackle), a service that 
provides distributed synchronization primitives. It defines five application cores (`GrackleLocks`, 
`GrackleSemaphores`, `GrackleNamespaces`, `GrackleWaitGroups`, `GrackleBarriers`) grouped under a single `Grackle` stub. 
Only the `GrackleLocks` core is shown in full below; the rest follow the same shape.

```yaml
go_code:
  output_package: github.com/evrblk/grackle/pkg/coreapis
  core_types_package: github.com/evrblk/grackle/pkg/corepb
cores:
  - name: GrackleLocks
    read_methods:
      - name: GetLock
        method_number: 1
        sharded: true
      - name: ListLocks
        method_number: 2
        sharded: true
      - name: ListLocksByLeaseId
        method_number: 3
        sharded: true
      - name: ListLockLeases
        method_number: 4
        sharded: true
      - name: ListLockLeasesByProcessId
        method_number: 5
        sharded: true
      - name: GetLockLease
        method_number: 6
        sharded: true
    update_methods:
      - name: AcquireLock
        method_number: 1
        sharded: true
      - name: ReleaseLock
        method_number: 2
        sharded: true
      - name: DeleteLock
        method_number: 3
        sharded: true
      - name: RunLocksGarbageCollection
        method_number: 4
        sharded: false
      - name: LocksDeleteNamespace
        method_number: 5
        sharded: true
      - name: CreateLockLease
        method_number: 6
        sharded: true
      - name: RefreshLockLease
        method_number: 7
        sharded: true
      - name: RevokeLockLease
        method_number: 8
        sharded: true
  # GrackleSemaphores, GrackleNamespaces, GrackleWaitGroups, and GrackleBarriers omitted for brevity
stubs:
  - name: Grackle
    cores:
      - GrackleLocks
      - GrackleSemaphores
      - GrackleNamespaces
      - GrackleWaitGroups
      - GrackleBarriers
```

Each method has a `method_number`. Like protobuf field numbers, method numbers identify a method on the wire and must 
be unique within a core and never change. `sharded: true` means the request is routed to a single shard by its shard key 
(see below); `sharded: false` is for cluster-wide or fan-out operations that target a shard by id explicitly. Read 
methods can also set `allow_read_from_followers: true` to allow serving them from follower replicas when slightly stale 
data is acceptable.

This config will generate:

* `GrackleClientApi` - interface for the stub `Grackle` which consists of methods from the `GrackleLocks`, 
  `GrackleSemaphores`, `GrackleNamespaces`, `GrackleWaitGroups`, and `GrackleBarriers` application cores. There are two 
  generated implementations of this interface:
  * `GrackleMonsteraStub` - the stub that wraps the Monstera Client. All the routing is done there. Constructed 
    with `NewGrackleMonsteraStub(monsteraClient)`.
  * `GrackleNonclusteredStub` - nonclustered stub. Constructed with `NewGrackleNonclusteredStub` together 
    with the generated `GrackleNonclusteredApplicationCoresFactory`. See 
    [Single Node Mode](/docs/single-node-mode.md) for more details.
* `GrackleLocksCoreApi`, `GrackleSemaphoresCoreApi`, etc. - interfaces for application cores (this is what you implement).
* `GrackleLocksCoreAdapter`, `GrackleSemaphoresCoreAdapter`, etc. - adapters for the corresponding application cores, 
  constructed with `NewGrackleLocksCoreAdapter`, `NewGrackleSemaphoresCoreAdapter`, etc.
* Request/response type aliases for every method (e.g. `GetLockRequest`, `GetLockResponse`) in the `output_package`. 
  These wrap your proto payloads (see below) and only improve code readability.

A stub can route requests to one or several application cores. Since cores tend to be small, it makes sense to group 
multiple of them in a single stub. The stub interface (`GrackleClientApi`) is essentially the union of the methods 
of its application cores, plus a `ListShards(applicationName string)` helper. Unlike the core methods, stub methods take 
a `context.Context` and the raw request/response types directly (sharded methods derive the shard key by calling 
`ShardKey()` on the request; unsharded methods take an extra `shardId string` argument).

Currently, Monstera codegen relies on several conventions in order to make it work with your code:

* Methods of an application core must have corresponding `*Request` and `*Response` structs in the 
  `go_code.core_types_package` package. For example, `GetLock` of the `GrackleLocks` core must have 
  `GetLockRequest` and `GetLockResponse` structs messages in `github.com/evrblk/grackle/pkg/corepb`.
* Those `*Request` and `*Response` structs must implement `encoding.BinaryMarshaler` and 
  `encoding.BinaryUnmarshaler` (`MarshalBinary()` / `UnmarshalBinary()`). The framework moves them around as opaque 
  bytes inside a generic envelope, so it does not need to know their concrete types. (Grackle generates these one-line 
  wrappers over the proto's `MarshalVT` / `UnmarshalVT` with a small helper tool of its own, but you can choose whatever
  you want).
* Every `*Request` of a sharded method must implement `ShardKey() []byte`. There are no annotations or reflection â€” you
  specify explicitly how to extract a shard key from each request, usually with one line of Go code.

Packages `output_package` and `core_types_package` should be different, otherwise it will cause name collision for 
`*Request`/`*Response` types.

You define one `*Request`/`*Response` pair per method. This is conceptually similar to a gRPC service declaration. 
The envelope exchanged over the Monstera client is the framework's own `rpc.Request` (carrying a `method_number`, 
the marshaled payload, and a request timestamp) and `rpc.Response` (carrying the marshaled payload and an application 
`rpc.Error`).

Here is an example `proto` file with all definitions necessary for the YAML config from above:

```proto
syntax = "proto3";

option go_package = "github.com/evrblk/grackle/pkg/corepb";
package com.evrblk.grackle.corepb;

message GetLockRequest {/*...*/}
message GetLockResponse {/*...*/}
message ListLocksRequest {/*...*/}
message ListLocksResponse {/*...*/}
message AcquireLockRequest {/*...*/}
message AcquireLockResponse {/*...*/}
message RunLocksGarbageCollectionRequest {/*...*/}
message RunLocksGarbageCollectionResponse {/*...*/}
// ... one *Request/*Response pair per method, for every core
```

That is all the proto you need: just the per-method request and response messages. No high-level containers, no shared 
`Error` field â€” those are provided by the framework's `rpc` package.

## Core API interfaces

The generated `*CoreApi` interface is what you implement. Its methods receive the request payload wrapped in one of the
framework's generic types and return the response payload wrapped accordingly. Codegen emits a type alias per method in 
the `output_package` so the signatures stay short (the generated code imports `github.com/evrblk/monstera/rpc` aliased 
as `mrpc`). For the config above:

```go
// sharded read
type GetLockRequest = mrpc.ReadRequest[*corepb.GetLockRequest]
type GetLockResponse = mrpc.ReadResponse[*corepb.GetLockResponse]

// sharded update
type AcquireLockRequest = mrpc.UpdateRequest[*corepb.AcquireLockRequest]
type AcquireLockResponse = mrpc.UpdateResponse[*corepb.AcquireLockResponse]

// an unsharded method uses the *Unsharded* wrapper for the request
type RunLocksGarbageCollectionRequest = mrpc.UpdateUnshardedRequest[*corepb.RunLocksGarbageCollectionRequest]
type RunLocksGarbageCollectionResponse = mrpc.UpdateResponse[*corepb.RunLocksGarbageCollectionResponse]
```

The request wrapper exposes the proto payload as `Payload` and the request timestamp as `Now` (Unix nanoseconds) â€” read 
`Now` instead of calling `time.Now()` so updates apply deterministically on every replica. The response wrapper carries 
the proto `Payload` and an optional `ApplicationError *mrpc.Error` that becomes the deterministic domain error returned 
to the caller. So the generated `GrackleLocksCoreApi` looks like:

```go
type GrackleLocksCoreApi interface {
    Snapshot() monstera.ApplicationCoreSnapshot
    Restore(reader io.ReadCloser) error
    Close()

    GetLock(req *GetLockRequest) (*GetLockResponse, error)
    ListLocks(req *ListLocksRequest) (*ListLocksResponse, error)
    AcquireLock(req *AcquireLockRequest) (*AcquireLockResponse, error)
    RunLocksGarbageCollection(req *RunLocksGarbageCollectionRequest) (*RunLocksGarbageCollectionResponse, error)
    // ... one method per read_method and update_method
}
```

The second returned value `error` is an internal error that killed the application core (not a domain error). See 
[Core Principles](/docs/core-principles.md) for more details.

What is left for you to implement:

* The application cores themselves (`GrackleLocksCoreApi`, `GrackleSemaphoresCoreApi`, etc. interfaces).
* `ShardKey() []byte` on every sharded `*Request` payload type.
* `MarshalBinary` / `UnmarshalBinary` on all request and response payload types.
